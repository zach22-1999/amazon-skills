# 6 问引导式流程：用 AskUserQuestion 一层一层问清楚

本文件是 `zach-seller-skill-creator` 阶段 0 的唯一执行脚本。**Claude 用它来问，用户不直接看它**。

---

## 为什么这样设计

老版本让用户一次性回答 6 道开放性问答题，卖家常常卡壳或敷衍。新版把每个主题拆成 1-3 层：
- **第 1 层**优先用 `AskUserQuestion`（选项题），帮用户用“点选”代替“凭空组织答案”
- **第 2-3 层**根据上一层的回答，继续用选项或一次短 free-text 追问
- 选项既含抽象大类，也内置亚马逊卖家常见的具体预设（Sorftime / 领星 / SIF / BCG 阈值等），用户能直接指认

目标：用户在约 5 分钟内，通过约 10 次点选 + 每题 1 段短简答，把 6 个维度全部说清楚。

---

## Claude 的使用方式

1. 顺序推进 `Q1 → Q6`，**一个主题走完再下一个**，不要跳。
2. 每个主题内部的层级也是顺序的。每层结束后，把用户答案记到临时笔记（Claude 自己记住即可，不要让用户复述）。
3. 用户选择 `Other` 时，接着问一个 free-text 让他补充具体内容。
4. 如果用户一上来贴长文，不要直接当成阶段 0 完成；要把内容拆回当前主题，补齐还缺的层级问题。
5. 每个主题结束时先做一次**逐题门禁**，不合格就当场补问，不要把模糊答案带到下一题。
6. Q1-Q6 全部完成后，再按文末的“总门禁判定清单”扫一遍；任一项不达标回到对应主题补问。
7. 全部通过后，用一段 bullet list 向用户复述 6 维答案，确认后再进入阶段 1。

**特殊情况**：用户说“我就想快速试试”时，只走 `Q1 + Q3 + Q6` 三个核心主题，其余记作待补，并明确告诉他：这会降低 skill 质量，后面如果结果偏空，需要回头补完。

## 第一部分：引导式问答脚本

### Q1 业务目标

#### L1 — 问题类型（AskUserQuestion，单选）

- **question**：这个 skill 主要帮你解决什么类型的业务问题？
- **header**：业务目标
- **multiSelect**：false
- **options**：
  - **自动化** — 把每周/每天都要做一次的流程自动化，比如定期周报、数据整理、模板填写。
  - **风险监控** — 监控库存告急、差评暴增、排名暴跌、广告失控等需要及时介入的情况。
  - **决策支持** — 辅助选品、定价、补货、广告调价、Listing 改版等判断型工作。
  - **团队交付** — 为运营、供应链、老板产出能直接用的交付物（周报、分析报告、飞书消息）。

#### L2 — 不做的损失（chat 简答）

用户选完 L1 后，接着问：

> 如果**不做这个 skill**，你会损失什么？能不能给个量化的感受，比如每周多花几小时、每月多浪费多少广告费、或者会错过什么机会？

收一句话或半段话即可。不接受“损失很多”“比较麻烦”这种空话，要追问成时间、金钱或机会损失。

#### Q1 逐题门禁

通过标准：
- 说清楚属于哪一类业务问题
- 说清楚“不做的代价”
- 至少有一个具体损失量纲：时间、钱、销量、机会，四选一

---

### Q2 现有做法

#### L1 — 自动化程度（AskUserQuestion，单选）

- **question**：你现在手动做这件事处于什么状态？
- **header**：现有做法
- **multiSelect**：false
- **options**：
  - **纯手动** — Excel / 肉眼对比 / 复制粘贴，没有工具辅助。
  - **用专业工具但还要串联** — 用了 Sorftime / 领星 / ERP / 飞书表格等专业工具，但各步骤之间要人肉搬数据。
  - **已经有半自动** — 有现成 Python 脚本、Excel 宏、或 n8n 工作流，但还需要手动触发和检查。
  - **全新流程** — 这事以前没系统做过，想直接从 skill 开始。

#### L2 — 卡点（chat 简答）

用户选完 L1 后，接着问（选“全新流程”可以跳过这题）：

> 你现在做一次大概花多久？最烦、最容易出错、或者最容易拖延的是哪一步？

目的是让他指认具体痛点。这一步的答案会直接进 `## 当前工作流（人工版）`，作为新 skill 的基线对照。

#### Q2 逐题门禁

通过标准：
- 说清楚当前是纯手动、半自动还是全新流程
- 如果不是全新流程，要说出至少一个耗时或痛点
- 不能只回答“现在也是这么做”“有 Excel”这种无信息答案

---

### Q3 具体步骤

这是最关键也最容易敷衍的一题，分 3 层。

#### L1 — 数据来源（AskUserQuestion，多选）

- **question**：这个流程会用到哪些数据来源？（多选）
- **header**：数据来源
- **multiSelect**：true
- **options**：
  - **Sorftime** — 市场 / 竞品 / 评论 / 类目研究工具。
  - **领星** — 店铺运营数据、订单、库存、利润等经营数据。
  - **SIF** — ASIN 关键词自然排名与广告占比分析。
  - **本地 Excel / CSV / TXT** — 用户上传或工作区已存的文件。
  - **飞书表格** — 数据在飞书表格里，需要读取或写回。
  - **人工粘贴 / 临时文本** — 用户直接把数据贴在聊天里。

如果用户选了 `Other`，追问是什么，常见可能是 Amazon 后台导出、apiclaw、邮件附件等。

#### L2 — 流程复杂度（AskUserQuestion，单选）

- **question**：这个流程的判断分支有多复杂？
- **header**：复杂度
- **multiSelect**：false
- **options**：
  - **线性流程** — 一路走到底，没有 if/else。
  - **少量分支** — 1-2 个判断（比如“如果异常就额外发飞书”）。
  - **复杂判断** — 多层条件、循环、回溯，需要仔细描述逻辑树。

#### L3 — 逐步骤描述（chat 简答，用脚手架引导）

用户选完 L1 + L2 后，给他一个填空式脚手架：

> 基于你选的这些，请把第 1 步到第 N 步具体写给我。每一步按这个格式说：
>
> ```text
> 第 X 步：[动作]
>   输入：[来自哪里？文件路径 / MCP 接口 / 上一步的输出]
>   输出：[产出什么？存到哪？]
>   判断：[如果 xxx 则 yyy，否则 zzz；没有判断就写“无”]
> ```
>
> 你刚才说的数据源是 {L1 多选结果}，复杂度是 {L2 结果}。按这个来写就行，不用追求完美，5-8 步通常就够。

**收到答案后，Claude 自检：**
- 每一步是否有“输入 / 输出 / 判断”三件？缺哪个就回头追问
- 如果用户只给了动作顺序，没给任何判断分支（但 L2 选的不是“线性流程”），要追问“你刚才说有分支，是在哪一步？”
- 如果用户只写“分析一下”“生成报告”，要追问“分析前读什么、按什么规则判断、最后输出到哪里”

#### Q3 逐题门禁

通过标准：
- 已知主要数据来源
- 步骤至少有 3 步，最好 5-8 步
- 每一步至少能看出输入和输出
- 非线性流程必须出现至少一个明确判断分支

---

### Q4 方法论

#### L1 — 判断框架（AskUserQuestion，多选）

- **question**：这个流程主要靠哪些判断框架或经验规则？（多选）
- **header**：方法论
- **multiSelect**：true
- **options**：
  - **阈值判断** — 先设阈值，超过/低于就判异常或触发动作。
  - **趋势对比** — 跟上周、上月、历史均值或基线对比，不只看单点数值。
  - **优先级排序** — 同时出现多个问题时，要按轻重缓急排序处理。
  - **异常归因** — 不是只报异常，还要判断更可能是哪类原因。
  - **例外排除** — 某些时段/人群/新品期要剔除，不纳入常规判断。
  - **BCG 健康阈值** — 用 BCG/ABC 分层或健康阈值做巡检。
  - **广告占比异常** — 广告占比、自然位、转化率联动看，不单看一个指标。
  - **新品期不纳入稳定判断** — 上架初期先做观察，不按成熟品标准判。
  - **节假日周单独处理** — Prime Day、黑五、圣诞等时间窗不直接和普通周硬比。

#### L2 — 最关键的规则（chat 简答）

用户选完 L1 后，接着问：

> 从你刚才选的这些里，挑最关键的 1-3 条规则写给我。最好写成“如果……则……”或“超过/低于多少就算异常”的句子。

理想答案示例：
- “排名变化 < 5 位不算异常，> 10 位才标红”
- “新品上架 30 天内不纳入稳定性判断”
- “同一个核心词连续 2 周下滑才升级为严重”

#### Q4 逐题门禁

通过标准：
- 至少有一个明确判断框架
- 至少有一条具体规则，不接受只有“看经验”“综合判断”
- 最好出现阈值、时间窗、优先级或例外条件之一

---

### Q5 调用方式

#### L1 — 触发方式（AskUserQuestion，单选）

- **question**：你希望这个 skill 平时怎么被触发？
- **header**：调用方式
- **multiSelect**：false
- **options**：
  - **一句话直接触发** — 我说一句话，它就自己跑。
  - **先问参数再执行** — 先问品牌、ASIN、时间范围、文件路径等，再跑。
  - **上传文件后自动识别** — 我把文件扔进去，它判断该怎么处理。

#### L2 — 业务入口（AskUserQuestion，多选）

- **question**：这个 skill 最常从哪些业务入口被叫起？（多选）
- **header**：业务入口
- **multiSelect**：true
- **options**：
  - **Sorftime 研究** — 从市场、竞品、评论、类目研究入口触发。
  - **领星经营数据** — 从经营数据、订单、库存、利润入口触发。
  - **SIF 排名诊断** — 从自然位、广告占比、关键词可见性入口触发。
  - **本地 Excel / CSV** — 从文件处理或表格整理入口触发。
  - **飞书表格 / 飞书消息** — 从飞书工作流入口触发。
  - **人工临时问题** — 老板或运营临时提一句，要马上出结果。

#### L3 — 真实触发话术（chat 简答）

用户选完 L1 + L2 后，接着问：

> 给我 2-3 句你真的会说出口的话。别写成说明书，按你平时跟我说话的口气写就行。

理想答案示例：
- “帮我看下 BCG 这周哪些词掉出前 20 了”
- “跑一下领星库存异常巡检”
- “我把这份 CSV 丢给你，你帮我整理成周报”

#### Q5 逐题门禁

通过标准：
- 已知默认触发方式
- 已知至少一个业务入口
- 至少有 2 句真实触发话术，不能只有“运行这个 skill”

---

### Q6 期望输出

#### L1 — 产物类型（AskUserQuestion，多选）

- **question**：这个 skill 跑完后，你最希望拿到什么类型的结果？（多选）
- **header**：期望输出
- **multiSelect**：true
- **options**：
  - **Markdown 报告** — 适合保存、复盘、发给团队。
  - **表格 / CSV** — 适合后续继续筛选、汇总、写回系统。
  - **飞书消息** — 适合异常提醒、日报、周报播报。
  - **异常清单** — 只列出要处理的问题项，不写长报告。
  - **行动建议** — 给出下一步怎么做的建议，不止报现状。

#### L2 — 落点（AskUserQuestion，多选）

- **question**：这些结果应该落到哪里？（多选）
- **header**：输出落点
- **multiSelect**：true
- **options**：
  - **本地目录** — 保存到工作区固定路径。
  - **飞书** — 发到飞书群、飞书文档或飞书表格。
  - **终端直接展示** — 不存文件，直接把结果回在终端里。

#### L3 — 命名规则与完成信号（chat 简答）

用户选完 L1 + L2 后，接着问：

> 最后补一句：文件怎么命名，或者做完之后你希望我怎么告诉你“已经完成”？

理想答案示例：
- “文件名带日期 + 品牌 + ASIN”
- “终端回一句：已完成，异常 3 个，报告已发飞书”
- “本地保存到固定目录，同时贴一段摘要”

#### Q6 逐题门禁

通过标准：
- 已知至少一种产物类型
- 已知至少一个输出落点
- 已知命名规则或完成信号中的至少一项

## 第二部分：总门禁判定清单

Q1-Q6 全部走完后，用这张清单再扫一遍：

- **Q1 业务目标**：不能只有“大词”，必须说清“不做的代价”
- **Q2 现有做法**：不能只说“手动做”，要能看出耗时、工具或痛点
- **Q3 具体步骤**：不能只有动作顺序，必须有输入 / 输出 / 判断逻辑
- **Q4 方法论**：必须至少有 1 条具体规则，不能只有“看经验”
- **Q5 调用方式**：必须有真实触发话术，不能只有抽象描述
- **Q6 期望输出**：必须同时知道“产物形式”和“落点”

任一项不达标，不得进入阶段 1。

## 第三部分：6 维答案到 SKILL.md 的映射

| 阶段 0 收集结果 | 映射到 SKILL.md 的哪里 |
|----------------|------------------------|
| Q1 业务目标 | `## 业务背景` 段（解释 why） |
| Q2 现有做法 | `## 当前工作流（人工版）` 段（基线对照） |
| Q3 具体步骤 | `## Skill 工作流（自动版）` 段（主干指令，转成编号步骤） |
| Q4 方法论 | `## 核心原则 / 踩坑规避` 段 |
| Q5 调用方式 | YAML `description` 字段 + `## 触发场景` 段 |
| Q6 期望输出 | `## 输出规范` 段 |

如果用户走的是“快速试试”路径，要把缺失题目标记为待补，不要伪装成已经采集完成。
